## 2. 优化前的主要问题

### 2.1 智能体与交互层

- 本地多模态模型生成温度偏高，指令理解在导航类短指令场景中稳定性不足。
- 泛化目标词，例如“饮料”，难以直接映射为用户真实偏好的具体物品。
- 智能体、视觉导航、语音交互之间的调用关系已经打通，但报告层面缺少“从输入到动作”的清晰说明。

### 2.2 云端部署与浏览器交互层

- 远程服务器无法直接使用用户本地摄像头和麦克风，原有本地设备假设不适合浏览器远程访问。
- 服务端需要兼容浏览器推流、浏览器音频播放和 WebSocket 双向通信，否则远程演示无法稳定运行。

### 2.3 视觉推理性能层

- 盲道导航与过马路模式存在明显卡顿，实时画面和语音反馈有积压。
- 多个视觉模型串行执行，导致某一模型变慢时会拖慢整条链路。
- 初期无法准确判断瓶颈是在网络传输、浏览器渲染，还是 GPU 推理本身。

## 3.4 性能埋点与瓶颈定位

本阶段主要对应提交 `30b0f47` 及当天持续调试。

为避免“凭感觉优化”，系统新增了两组关键埋点：

- 相机链路埋点 `[PERF]`：记录 `recv`、`overwrite`、`display_delay`、`yolo_submit`、`yolo_busy_skip`、`yolo_queue`、`yolo_proc`、`yolo_e2e` 等指标，每 5 秒打印一次。
- 盲道流程细分埋点 `[BLINDPATH-TIMING]`：把 `process_frame` 拆成 `yolo_seg`、`obstacle`、`rest`、`total` 四段单独计时。

通过这套埋点，最终得到两个关键结论：

1. 浏览器链路不是主瓶颈，`display_delay` 实测仅约 5ms 量级。
2. 卡顿主要来自推理侧，尤其是盲道主分割模型，而不是网络传输或前端显示。

## 3.5 推理链路性能优化

### 3.5.1 相机显示线程与 YOLO 推理解耦

- `app_main.py` 中使用独立显示线程和独立 YOLO 线程，显示线程优先把最新 JPEG 直接转发给浏览器。
- 当 YOLO 线程忙时，新帧不排长队，而是覆盖旧待处理帧，只保留“最新一帧待推理”，避免排队延迟累积。
- 这样做牺牲了部分帧的逐帧推理，但显著提升了“看起来实时”的用户体验。

### 3.5.2 YOLO 推理参数轻量化

在以下模块中统一引入 `imgsz=320` 与 `half=True`：

- `workflow_blindpath.py`
- `workflow_crossstreet.py`
- `trafficlight_detection.py`
- `obstacle_detector_client.py`
- 相关 YOLO 推理入口

其作用是：

- 用更小输入尺寸换取更快推理速度。
- 在支持 FP16 的 GPU 上使用半精度推理，降低显存和算力压力。

### 3.5.3 障碍物检测异步化

- 盲道导航中最重的辅助模型之一是 YOLO-E 障碍物检测器。
- 原先障碍物检测和主分割模型共用主流程，导致某些帧会同时执行两次重推理，严重拖慢主循环。
- 现在通过后台线程 `_obstacle_worker` 异步处理障碍物，只把最新图像提交给后台线程，并复用最近一次检测缓存。

这一改动带来的直接结果是：

- 障碍物分支的计时从秒级下降到约 `0-48ms`。
- 主线程不再因为 YOLO-E 阻塞而整体停顿。

### 3.5.4 语音反馈策略压缩

- 对组合语音提示只播第一段，避免长语音堆积导致“视觉早就变化了，语音还在播旧指令”的问题。
- 强制提高障碍物语音优先级，减少次要导航语句抢占播报通道。

## 4. 关键实验结论

根据本轮调试记录，得到如下结论：

| 场景 | 现象 | 结论 |
| --- | --- | --- |
| 浏览器显示链路 | `display_delay` 约 5ms | 网络与前端渲染不是主瓶颈 |
| 盲道导航模式 | `yolo_proc` 约 2000-7500ms | 主分割模型过重，是核心卡点 |
| 过马路模式 | `yolo_proc` 约 600-1900ms | 同样受分割模型规模影响，但负担略轻 |
| 障碍物检测分支 | 异步化后 `obstacle` 约 0-48ms | 后台线程方案有效 |
| 关闭障碍物检测后 | 盲道主流程仍约 2000ms | 根因不在 YOLO-E，而在主分割模型 |

也就是说，本轮优化已经证明：

- 云服务器卡顿不是“上云就必然慢”。
- 网络也不是主要问题。
- 真正需要替换的是当前的盲道分割模型规模。

## 5. 根因定位：当前主模型过重

调试过程中确认，当前 `model/yolo-seg.pt` 实际对应的是 `YOLOv8x-seg`：

- 参数量：`71.7M`
- 计算量：`329 GFLOPs`
- 层数：`231`

对于 Tesla P4 这类显存和算力都比较有限的服务器 GPU，`YOLOv8x-seg` 做实时盲道分割明显偏重，因此即使完成了 FP16、线程解耦和障碍物异步化，主分割耗时仍然停留在秒级。

与之对应，障碍物模型 `yoloe-11l-seg.pt` 也较重，但已经通过异步化把阻塞影响隔离出去。最终剩下的瓶颈只剩主分割模型本身。

## 6. 模型更新与轻量化训练方案

### 6.1 当前类别映射

当前导航分割模型的目标类别映射已经统一为：


这与现有导航逻辑保持一致：

- 类别 0：斑马线 / road_crossing
- 类别 1：盲道 / blind_path

### 6.2 新训练数据来源

### 6.3 新训练脚本

新增 `train_yolov8n_seg.py`，完成以下工作：

1. 从 Roboflow 下载两个数据集。
2. 自动合并 `train/valid/test` 目录。
3. 自动重映射类别 ID 到统一标签空间。
4. 以 `yolov8n-seg.pt` 为预训练权重进行迁移学习。
5. 训练结束后自动输出 `model/yolo-seg-nano.pt`。

### 6.4 轻量化替换目标

本轮模型更新的目标不是简单“再训一次”，而是把原来的大模型替换为轻量级 `YOLOv8n-seg`，从根源解决秒级分割耗时问题。

预期效果是：

- 盲道/斑马线分割从秒级下降到更适合实时导航的级别。
- 与已经完成的线程解耦、FP16 和异步障碍物检测形成叠加效果。
- 为后续封装独立 Web API 模型服务打下基础。

需要说明的是，当前训练脚本和数据集准备已经完成，但本轮文档只描述“模型更新路径”和“预期收益”，最终精度与速度仍需以训练后的实测结果为准。

## 8. 独立训练模型的 Web API 封装方案

### 8.1 封装目标

本项目中的“独立训练模型”指的是重新训练得到的轻量化盲道/斑马线分割模型，即 `yolo-seg-nano.pt`。为满足课程对“模型按 Web API 方式封装”的要求，推荐采用与现有主服务一致的 `FastAPI` 风格，对外暴露标准 HTTP 接口，便于：

- 使用 Postman 直接上传图片并获取推理结果。
- 让网页前端、主导航服务和第三方程序共用同一套模型能力。
- 在后续更换模型版本时保持接口不变，只替换模型文件。

从工程实现角度，第一阶段可以把模型接口直接挂载到现有主服务中，统一使用 `/api/model/blindpath/*` 路径；后续若需要单独拆分为独立进程或独立容器，也只需保持接口契约不变即可。

### 8.2 推荐接口清单

| 接口 | 方法 | 作用 | 是否适合 Postman |
| --- | --- | --- | --- |
| `/api/model/blindpath/health` | `GET` | 检查模型服务是否在线 | 是 |
| `/api/model/blindpath/meta` | `GET` | 返回模型名称、类别映射、默认参数 | 是 |
| `/api/model/blindpath/predict` | `POST` | 上传单张图片并返回分割结果 | 是 |

若直接使用 `FastAPI` 默认配置，服务启动后还可以通过 `/docs` 自动生成 Swagger 页面，便于教师验收和接口联调展示。

### 8.3 核心推理接口定义

推荐将核心接口定义为：

- 接口地址：`/api/model/blindpath/predict`
- 请求方法：`POST`
- 请求类型：`multipart/form-data`

推荐请求字段如下：

| 字段名 | 类型 | 是否必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `image` | File | 是 | 无 | 待检测图片，支持 jpg、jpeg、png |
| `conf` | Float | 否 | `0.25` | 置信度阈值 |
| `iou` | Float | 否 | `0.45` | NMS 阈值 |
| `imgsz` | Integer | 否 | `320` | 推理输入尺寸 |
| `device` | String | 否 | `cuda:0` | 推理设备，CPU 调试时可传 `cpu` |
| `return_polygon` | Boolean | 否 | `true` | 是否返回分割多边形坐标 |

推荐返回示例：

```json
{
	"success": true,
	"model_name": "yolo-seg-nano",
	"model_path": "model/yolo-seg-nano.pt",
	"classes": {
		"0": "road_crossing",
		"1": "blind_path"
	},
	"image_size": {
		"width": 1280,
		"height": 720
	},
	"inference_ms": 86.4,
	"detections": [
		{
			"class_id": 1,
			"class_name": "blind_path",
			"confidence": 0.92,
			"bbox_xyxy": [148.2, 102.5, 1080.6, 695.4],
			"polygon": [[182, 694], [225, 615], [302, 543], [512, 448], [874, 486], [1058, 688]],
			"mask_area_ratio": 0.183
		}
	]
}
```

其中：

- `class_id` 和 `class_name` 直接对应训练脚本中统一后的类别映射。
- `bbox_xyxy` 便于常规检测框显示。
- `polygon` 便于前端重绘分割轮廓或用于后续导航几何计算。
- `mask_area_ratio` 便于快速判断目标占画面比例，为导航策略提供简化特征。

### 8.4 建议返回状态码

| 状态码 | 含义 | 典型场景 |
| --- | --- | --- |
| `200` | 调用成功 | 已返回推理结果 |
| `400` | 请求参数错误 | 未上传图片，或字段格式不正确 |
| `415` | 文件格式不支持 | 上传了非图片文件 |
| `503` | 模型未就绪 | 模型文件不存在，或服务启动时加载失败 |
| `500` | 推理失败 | 推理过程中发生异常 |

### 8.5 Postman 调用方式

服务启动后，可以直接使用 Postman 对上述接口进行标准化测试。推荐测试步骤如下：

1. 打开 Postman，新建一个 `POST` 请求。
2. 请求地址填写 `http://127.0.0.1:8081/api/model/blindpath/predict`；若在服务器环境测试，则把 `127.0.0.1` 替换为服务器公网 IP。
3. 在 `Body` 中选择 `form-data`。
4. 添加字段 `image`，类型选择 `File`，上传一张盲道或斑马线测试图片。
5. 可选添加 `conf=0.25`、`iou=0.45`、`imgsz=320`、`device=cuda:0`、`return_polygon=true`。
6. 点击发送，检查返回是否为 `200`，并确认结果中包含 `classes`、`inference_ms` 和 `detections` 字段。

Postman 中可直接保存以下典型测试组合作为课程报告附件：

- 盲道清晰图像测试：验证 `blind_path` 能否稳定输出多边形。
- 斑马线图像测试：验证 `road_crossing` 类别是否按统一映射返回。
- 空场景图像测试：验证无目标时 `detections` 返回空数组。
- 错误输入测试：验证未上传图片或文件类型错误时的异常处理。

### 8.6 可直接写入报告的接口封装表述

为满足“独立训练模型需按 Web API 方式封装”的要求，本项目计划将重新训练得到的 `YOLOv8n-seg` 轻量模型封装为标准 HTTP 推理服务。接口采用 `FastAPI` 实现，提供健康检查、模型元信息查询和单图推理三个核心接口，其中推理接口使用 `multipart/form-data` 方式接收图片，并以 JSON 形式返回类别、置信度、分割多边形和推理耗时等结果。该封装方式既能与现有 Web 系统无缝集成，也便于使用 Postman 对服务接口进行独立测试和结果留档。
